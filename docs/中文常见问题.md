# rt-llm-proxy 中文 FAQ

## 功能相关

### Q: rt-llm-proxy 是什么？

**A:** 这是一个用 Go 编写的实时 LLM 代理。浏览器通过 WebRTC 连接，代理将音频转发给 LLM（如 Gemini、豆包），获得实时语音回复。支持托管 API 或自托管的 ASR→LLM→TTS 管道。

### Q: 支持哪些 LLM 提供商？

**A:** 目前支持四种：

| 提供商 | 说明 | 状态 |
|---|---|---|
| Gemini | Google 的 Gemini Live API | ✅ 完全支持 |
| 豆包 | Volcengine 的端到端实时语音 | ✅ 完全支持 |
| Cascade | 自托管 ASR→LLM→TTS | ✅ 完全支持 |
| Loopback | 测试用的假提供商 | ✅ 仅用于负载测试 |

### Q: 级联（Cascade）是什么？为什么需要它？

**A:** 级联是一个自托管的 ASR→LLM→TTS 管道：

- **ASR（语音识别）** — RealtimeSTT（Silero VAD + Whisper）
- **LLM（大语言模型）** — vLLM（运行 Qwen 等开源模型）
- **TTS（语音合成）** — XTTS（从文本生成语音）

**为什么使用：**
- 完全自主控制（不依赖云 API）
- 可以自定义模型和工作流
- 支持"DJ 用例"（注入实际音乐轨道）
- 延迟可控（所有组件在 GPU 主机的 LAN 内）

### Q: 能否在 CPU 主机上运行级联？

**A:** 理论上可以，但不推荐。级联需要 GPU 加速：

- **ASR** — 可以用 CPU，但慢
- **LLM** — 需要 GPU，否则极慢（Qwen 7B 在 CPU 上不实用）
- **TTS** — 可用 CPU，但延迟高

**建议配置**：NVIDIA L20 或更高（24GB VRAM）。在 Volcano 引擎、AWS、或本地可用。

### Q: 支持中文吗？

**A:** 完全支持！

- **ASR**（语音识别）— RealtimeSTT 支持中文
- **LLM** — Qwen 3.5 原生支持中文
- **TTS**（语音合成）— XTTS 支持中文（`-cascade-tts-lang zh-cn`）

在中文环境下运行示例：

```bash
docker compose -f docker-compose.yml \
               -f docker-compose.cascade.yml \
               up --build

# 访问 http://<PUBLIC_IP>:8080/demo/?model=cascade
# 设置语言为中文
```

## 部署和安装

### Q: 最快开始的方式是什么？

**A:** 一行命令：

```bash
export GEMINI_API_KEY=your_key && go run ./cmd/proxy && open http://localhost:8080/demo/
```

需要 Go 1.25+ 和 libopus。更详细的步骤见[中文部署指南](中文部署指南.md)。

### Q: 我没有 GPU，能运行代理吗？

**A:** **完全可以**——但只能使用 Gemini 或豆包（云提供商），不能运行自托管级联。

```bash
# 这就可以了
export GEMINI_API_KEY=...
go run ./cmd/proxy -addr :8080
```

级联需要 GPU。

### Q: 如何在国内使用？

**A:** 几个要点：

1. **Go 依赖加速**（必须）：
   ```bash
   export GOPROXY=https://goproxy.cn,direct
   ```

2. **Docker 镜像加速**（如果用 Docker）：
   ```bash
   docker compose -f docker-compose.yml \
                  -f docker-compose.cn.yml up --build
   ```

3. **Gemini API 访问** — 可能需要 VPN（Google API 的网络限制）

4. **豆包** — 本地化 API，无需 VPN，直接使用即可

### Q: Docker 构建很慢，怎么办？

**A:** 中国用户配置 Go 代理：

**方法 1**：在 `.env` 中添加
```bash
GOPROXY=https://goproxy.cn,direct
```

**方法 2**：使用 CN 叠加层
```bash
docker compose -f docker-compose.yml -f docker-compose.cn.yml up --build
```

## 性能和容量

### Q: 一台主机最多支持多少个并发会话？

**A:** 取决于提供商和硬件：

| 场景 | 容量 | 限制 |
|---|---|---|
| 仅 Gemini / 豆包 | ~600–1000 / 16 核 | CPU（Opus 编解码）|
| 级联（自托管） | ~50–100 / GPU | 显存（LLM + TTS） |

基准数据：
- **单核最多** ~107 个会话（仅 Opus）
- **16 核 CPU**（带 SRTP/DTLS 开销） ~600–1000
- **L20 GPU**（Qwen + XTTS） 视负载 ~50–100

### Q: 如何优化性能？

**A:** 三个杠杆：

1. **自适应 Opus 复杂度**（推荐）：
   ```bash
   go run ./cmd/proxy -adaptive sessions
   ```
   自动在高负载时降低编码质量以保持实时性。

2. **手动降低 Opus 复杂度**：
   ```bash
   go run ./cmd/proxy -opus-complexity 5  # 默认 10
   ```
   从 0–10，越低 CPU 越少，质量越差。

3. **增加硬件**：
   - 更多 CPU 核心（16→32 核）
   - 更好的 GPU（L20 → L40）

### Q: 延迟有多低？

**A:** 端到端延迟约 **200–400ms**（视提供商和网络）：

- **WebRTC 建立** ~ 100ms
- **ASR（仅语音识别）** ~ 100–200ms（Whisper）
- **LLM 首个令牌** ~ 200–500ms（Qwen）
- **TTS 合成** ~ 100–200ms（XTTS）
- **网络往返** ~ 10–50ms

级联 + 自托管给你最低延迟（不依赖云 API）。

### Q: 帧丢失或卡顿怎么办？

**A:** 检查两个指标：

1. **管理端点**：
   ```bash
   curl http://localhost:6060/stats | jq '.frames_late_30ms'
   ```
   如果 >5%，启用自适应复杂度。

2. **启用自适应**：
   ```bash
   go run ./cmd/proxy -adaptive sessions
   ```

3. **检查 CPU 使用率**：
   ```bash
   top  # 查看 %CPU 列
   ```

## 配置和集成

### Q: 如何启用速率限制？

**A:** 需要 Redis：

```bash
# 1. 启动 Redis
redis-server &

# 2. 运行代理
go run ./cmd/proxy -redis localhost:6379 -rl-max 10 -rl-window 1m
```

这限制每个客户端 IP 每分钟最多 10 个新会话。

如果 Redis 故障，代理仍然工作（失败开放）。

### Q: 如何看转录？

**A:** 三种方式：

1. **实时输出**（开发用）：
   ```bash
   go run ./cmd/proxy -sidechannel stdout
   ```
   转录打印到终端。

2. **Kafka**（生产用）：
   ```bash
   docker compose -f docker-compose.yml \
                  -f docker-compose.kafka.yml up --build
   
   # 消费转录
   docker compose exec kafka kafka-console-consumer.sh \
     --bootstrap-server localhost:9092 \
     --topic transcripts
   ```

3. **数据通道**（浏览器）：
   演示页面实时显示转录。

### Q: 如何自定义 LLM 系统提示？

**A:** 对于级联：

```bash
go run ./cmd/proxy \
  -cascade-system "You are a friendly DJ who plays songs based on mood."
```

对于 Gemini / 豆包 — 由提供商定义，不可自定义。

### Q: 我想在 LLM 回复中注入音乐，怎么做？

**A:** 这是级联的"输出混合接缝"用例：

```go
cascade.New(ctx, cascade.Config{
    OnLLMToken: func(token, accumulated string) (string, bool) {
        // 检测用户是否要求播放歌曲
        if id := detectSongIntent(accumulated + token); id != "" {
            go playTrack(id)  // 触发外部播放
            return "", true   // 从 TTS 输出中删除此令牌
        }
        return "", false  // 正常通过
    },
})

// 会话中切换源
cascade.SetAudioSource(NewMP3Source("track-42.mp3"))
```

详见[中文指南](中文指南.md#两个业务接缝)。

## 故障排查

### Q: 浏览器无法连接，得到 "Failed to connect"

**A:** 检查清单：

1. **代理运行？**
   ```bash
   curl http://localhost:8080/stats
   ```
   应返回 JSON。

2. **防火墙？**
   ```bash
   # 开放 UDP 媒体端口
   sudo ufw allow 10000:60000/udp
   ```

3. **日志？**
   ```bash
   docker compose logs -f rt-llm-proxy | grep -i error
   ```

4. **API 密钥有效？**
   ```bash
   echo $GEMINI_API_KEY  # 应该不为空
   ```

### Q: API 限流或 429 错误

**A:** 两个原因：

1. **速率限制满** — Redis 策略拒绝了请求：
   ```bash
   # 增加限制
   go run ./cmd/proxy -rl-max 50 -rl-window 1m
   ```

2. **提供商 API 限流** — Gemini / 豆包 API 限制：
   ```bash
   # 使用电路断路器
   go run ./cmd/proxy -model-cb true -model-cb-open-after 5
   ```

### Q: 级联连不上，得到 502 错误

**A:** 侧车未运行或网络隔离：

```bash
# 1. 检查所有容器运行
docker compose ps

# 2. 验证侧车可达
docker compose exec rt-llm-proxy curl http://realtimestt:9000/health

# 3. 查看侧车日志
docker compose logs realtimestt -n 50  # 最后 50 行
docker compose logs vllm -n 50
docker compose logs xtts -n 50

# 4. 重启堆栈
docker compose down && docker compose up --build
```

### Q: 级联延迟太高，TTFA (时间到首音频) 超过 1 秒

**A:** 三个优化：

1. **启用推测性 LLM**（默认启用）：
   ```
   # 当 ASR 部分看起来完整时，在最终到达前启动 LLM
   # 自动工作，无需配置
   ```

2. **使用更快的模型**：
   ```bash
   docker compose down
   
   # 使用较小的 LLM（如 Qwen 1.5B 而非 7B）
   export CASCADE_LLM_MODEL=Qwen1.5B
   docker compose up --build
   ```

3. **预加载模型权重**：
   ```bash
   export QWEN_MODEL_PATH=/local/path/to/model
   docker compose up --build
   ```
   避免运行时从 HuggingFace 下载。

### Q: 内存泄漏或随时间 CPU 增加

**A:** 检查：

1. **Goroutine 泄漏**（级联特定）：
   ```bash
   curl http://localhost:6060/debug/pprof/goroutine?debug=1 | tail -20
   ```
   应该随着会话关闭而减少。

2. **缓冲区回收**（Opus）：
   - P6 优化应该零分配。
   - 如果 GC 频繁，检查日志中的"allocs"。

3. **Kafka 备压**：
   ```bash
   # 检查转录事件是否堆积
   docker compose exec kafka kafka-consumer-groups.sh \
     --bootstrap-server localhost:9092 --list
   ```

## 部署和运维

### Q: 如何在生产环境中运行？

**A:** 最少配置：

1. **前置反向代理**（如 Nginx）：
   - 终止 TLS
   - 负载均衡（多实例）
   - 注入 `X-Forwarded-For`

2. **使用 TURN / SFU**（如 LiveKit）：
   - NAT 穿透
   - 水平扩展
   - 会话故障转移

3. **启用 Redis**：
   - 速率限制
   - 跨实例共享计数

4. **启用 Kafka**：
   - 转录持久化
   - 审计日志
   - 跨节点重连支持

```bash
docker compose -f docker-compose.yml \
               -f docker-compose.redis-kafka.yml up --build
```

### Q: 可以在 Kubernetes 中运行吗？

**A:** 部分可以。**代理本身没问题，但 WebRTC 媒体困难：**

- ✅ 可以：HTTP(S) 控制平面、Offer 端点、管理 API
- ❌ 困难：UDP 媒体（需要 `hostNetwork`、主机选择器）

**建议**：部署在单主机或运行在 Kubernetes 外，前置 TURN / SFU 处理媒体路由。

### Q: 如何监控生产系统？

**A:** 设置这些告警：

1. **会话失败率**：
   ```bash
   curl http://localhost:6060/stats | jq '.failed_dials'
   ```
   告警如果 > 5% 失败率。

2. **帧延迟（SLO）**：
   ```bash
   curl http://localhost:6060/stats | jq '.frames_late_30ms'
   ```
   告警如果 > 5% 超过 30ms。

3. **Goroutine 泄漏**：
   ```bash
   curl http://localhost:6060/debug/pprof/goroutine?debug=1 | wc -l
   ```
   告警如果 > 预期基线 × 2。

4. **内存使用**：
   ```bash
   curl http://localhost:6060/debug/pprof/heap
   ```
   用 pprof 分析趋势。

### Q: 如何安全部署？

**A:** 关键点：

1. **认证**（Bearer 令牌）：
   ```bash
   curl -H "Authorization: Bearer your_token" \
        http://localhost:8080/offer -d "..."
   ```

2. **限制管理端口**：
   ```bash
   # 仅本地
   go run ./cmd/proxy -admin 127.0.0.1:6060
   ```

3. **Redis ACL**：
   ```bash
   # 设置强密码
   redis-cli CONFIG SET requirepass strong_password
   ```

4. **Kafka SASL**：
   ```bash
   # 启用 SASL/SCRAM
   export KAFKA_SASL_USERNAME=user
   export KAFKA_SASL_PASSWORD=password
   ```
